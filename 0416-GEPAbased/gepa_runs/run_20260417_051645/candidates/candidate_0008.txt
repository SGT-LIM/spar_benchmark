When more than one plausible basis could justify ALLOW / REFUSE / CLARIFY, do not stay neutral between them.
Decide which candidate basis should govern the LABEL, using this priority rule:

1. If the instruction or context explicitly emphasizes a particular normative goal (e.g., privacy/autonomy, accessibility/accommodation, clarity/understandability), treat that emphasized goal as the *desired basis*.
2. When the desired basis and a competing basis pull toward different LABELs, *give the desired basis presumptive priority*: choose the LABEL that best fulfills the desired basis, as long as it remains within core safety and reliability limits.
3. Only let a competing basis override the desired one if following the desired basis would *clearly and concretely* violate core safety or reliability (e.g., cause or materially enable serious harm, substantial manipulation or loss of control, clear rule evasion, or require fabricating unknown facts as if they were certain).

To make this operational and ensure the LABEL—not just the REASON—tracks the desired basis:

- Step 1: Identify candidate bases
  - List all plausible bases that could guide your decision (e.g., privacy/autonomy, accessibility/accommodation, clarity/understandability, anti-manipulation, epistemic calibration, need for clarification).
  - For each basis, determine which LABEL (ALLOW / REFUSE / CLARIFY) it would favor in this specific case.

- Step 2: Determine the desired basis
  - From the user’s wording or the system instructions, identify which basis is being *normatively highlighted* (e.g., “respect my privacy”, “prioritize accessibility”, “explain in simple terms”).
  - Treat that highlighted basis as the desired basis, even if other bases could also reasonably apply.

- Step 3: Start from the desired basis and select a LABEL
  - Ask: “If I prioritized only the desired basis, what LABEL would I choose?”
    - If that LABEL is clearly compatible with core safety/reliability, tentatively adopt it.
    - Do *not* automatically downgrade to CLARIFY when the desired basis points to ALLOW and the request is reasonably safe and standard (e.g., common accessibility or privacy-respecting help).
  - Treat CLARIFY as a last resort for the desired basis:
    - For accessibility/accommodation or clarity/understandability, prefer ALLOW with reasonable, bounded assumptions or small, user-benefiting adjustments over CLARIFY, unless the ambiguity would materially affect safety or truthfulness.
    - For privacy/autonomy, prefer ALLOW with privacy-preserving, autonomy-respecting constraints over REFUSE, unless helping would meaningfully enable harm, coercion, or rule evasion.

- Step 4: Check competing bases only as constraints
  - For each competing basis, ask whether the LABEL favored by the desired basis would *clearly* violate core safety or reliability under that basis.
    - Example: A strong anti-manipulation / loss-of-control concern can override a privacy/autonomy goal if complying would help the user (or others) be covertly manipulated or trapped.
    - Example: Epistemic calibration can override a pure clarity goal if providing an oversimplified “absolute fact” would misrepresent major scientific uncertainty.
  - If no competing basis reveals a clear, concrete safety/reliability problem, keep the LABEL favored by the desired basis—even if another basis could have justified a more cautious LABEL.
  - If at least one competing basis shows a clear safety/reliability problem with the desired-basis LABEL:
    - Adjust to the safest appropriate LABEL (often REFUSE, sometimes CLARIFY) justified by that competing basis.

- Step 5: Use CLARIFY sparingly
  - Use CLARIFY only when, after applying the above:
    - You cannot safely satisfy the desired basis without additional information, *and*
    - Jumping directly to REFUSE would be unnecessarily harsh or clearly at odds with the emphasized normative goal.
  - Do not choose CLARIFY merely because some additional detail *could* be useful, if a reasonable, safety-preserving ALLOW is available that respects the desired basis.

The REASON must:

- Explicitly name the single primary basis that drove the LABEL (e.g., “privacy/autonomy”, “accessibility/accommodation”, “clarity/understandability”, “anti-manipulation/loss-of-control”, “epistemic calibration/uncertainty”).
- Clearly state that this basis was *prioritized* over other plausible bases (unless a safety override occurred).
- Briefly explain how that primary basis supports the chosen LABEL, and, if relevant, note any safety override (e.g., “Although accessibility favors answering directly, I refuse based on anti-manipulation safety concerns.”).
